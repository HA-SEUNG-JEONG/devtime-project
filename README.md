# DevTime 프로젝트 명세서

## 1. 메인 페이지 (비로그인)

### 1.1 기본 기능

- DevTime의 모든 기능은 로그인을 해야만 이용 가능합니다.
- 타이머 관련 기능(타이머 재생, 할 일 목록, 초기화) 버튼을 클릭하면 로그인이 필요함을 안내하는 모달이 화면에 표시됩니다.
- GNB 메뉴(대시보드, 랭킹)를 누르면 로그인이 필요함을 안내하는 모달이 화면에 표시됩니다.
- GNB의 로그인, 회원가입 메뉴를 클릭하면 각각 로그인 페이지와 회원가입 페이지로 이동합니다.
- GNB의 DevTime 로고를 누르면 메인(타이머 페이지) 페이지로 이동합니다.

---

## 2. 로그인 페이지

로그인 정보를 확인해 유저 정보를 인증하고, 권한이 필요한 페이지에 접근할 수 있도록 인가 권한을 부여합니다.

### 2.1 기본 기능

- 로그인에는 이메일 주소와 비밀번호 정보가 필요합니다.
- 2개의 입력창을 통해 이메일 주소와 비밀번호 정보를 각각 입력받습니다.
- 이메일 주소 입력창은 입력값이 그대로 화면에 표시됩니다.
- 비밀번호 입력창은 입력 시 ⚫️로 값이 가려져서 표시됩니다.
- 로그인에는 `POST /api/auth/login` API를 사용합니다.
- 요청의 바디(Body)에는 입력 받은 이메일과 비밀번호 정보를 각각 `email`, `password` 속성의 문자열(String) 형태 값으로 반영해 요청을 보냅니다.
- 로그인하기 버튼 아래에 '회원가입' 텍스트를 클릭하면 회원가입 페이지로 이동합니다.

### 2.2 유효성 검증

#### 이메일 유효성 검사 규칙

- 이메일 주소는 공백일 수 없습니다.
- 이메일 주소 형식(e.g. `example@naver.com`)을 따라야 합니다.

#### 비밀번호 유효성 검사 규칙

- 비밀번호는 공백일 수 없습니다.
- 총 길이가 8자 이상이어야 합니다.
- 영문과 숫자의 조합으로 구성돼야 합니다.

#### 유효성 검증 결과에 따른 UI 동작

- 입력 창을 최초로 활성화하기 전에는 기본 상태를 유지합니다.
- 입력 창이 한 번 이상 활성화되고, 입력한 정보가 유효성 검증을 통과하지 못하면:
  - 입력창의 경계가 빨간색으로 바뀝니다.
  - 아래와 같은 경고 메시지가 입력창 아래에 표시됩니다:
    - **아이디**: 이메일 형식으로 작성해 주세요.
    - **비밀번호**: 비밀번호는 8자 이상, 영문과 숫자 조합이어야 합니다.
- 이메일과 비밀번호의 유효성 검사가 통과돼야만 로그인 버튼이 활성화되고, 둘 중 하나라도 미입력되거나 유효성 검사 기준을 통과하지 못하면 로그인 시도를 할 수 없습니다.

### 2.3 실패/성공 처리

#### 로그인 실패

- 로그인에 실패할 경우 '로그인 정보를 다시 확인해 주세요'라는 알림 메시지를 브라우저에 띄워줍니다.
- 로그인 실패 알림 메시지를 끄면 포커스가 이메일 입력 창으로 자동 이동합니다.

#### 로그인 성공

- 로그인 성공 시, 응답 객체 중 `isFirstLogin` 값을 통해 프로필 입력이 되어있는 상태인지를 파악하여 아래와 같이 분기 처리합니다:
  - **회원가입 후 첫 로그인인 경우** (`isFirstLogin: true`): 프로필 설정 페이지로 이동합니다.
  - **회원가입 후 첫 로그인이 아닌 경우** (`isFirstLogin: false`): 타이머 페이지로 이동합니다.
- 로그인에 성공했다면, 뒤로 가기로 다시 로그인 페이지에 접근하지 못하도록 제어합니다.

### 2.4 인증-인가 정책

- JWT 토큰 기반 인증 방식을 사용하여 인증/인가를 처리합니다.
- 로그인 성공 시, 응답 객체 중 `accessToken`과 `refreshToken` 속성의 값으로 토큰 정보가 발급됩니다.
- **Access 토큰**의 유효 시간은 **1시간**입니다.
- **Refresh 토큰**의 유효 시간은 **10일(240시간)**입니다.
- Access 토큰 만료 시, `POST /api/auth/refresh` API를 사용해 Refresh 토큰 정보를 전달하여 Access 토큰을 갱신합니다.
- Refresh 토큰 만료 시 서비스에서 로그아웃되며, 재로그인을 유도하기 위해 로그인 페이지로 이동합니다.

### 2.5 중복 로그인 처리

- 두 개 이상의 멀티 디바이스에서 중복 로그인이 불가능합니다.
- 하나의 기기(A)에 먼저 로그인이 되어있는 상태에서 새로운 기기(B)에서 로그인 시도가 이루어지면 먼저 로그인이 되어있던 기기(A)로 발급됐던 Refresh 토큰이 만료 처리하여 기존 기기에서 강제 로그아웃 시킵니다.
- 중복 로그인 시 먼저 발급된 Refresh 토큰을 만료시키기 위하여 서버에서는 Refresh 토큰 정보를 별도의 세션 DB에 저장하여 관리합니다.
- 동일 유저에게 두 개 이상의 Refresh 토큰이 발행될 시, 먼저 발급됐던 토큰 정보를 삭제하는 식으로 중복 로그인에 대응합니다.
- Refresh 토큰 강제 만료는 서버 로직으로 처리되며, 클라이언트에서는 `POST /api/auth/login` API 요청의 응답으로 반환되는 `isDuplicateLogin` 값을 사용해 중복 로그인 여부를 파악할 수 있습니다.
- `isDuplicateLogin` 값에 따른 처리:
  - **`true`**: 중복 로그인된 다른 기기가 존재하는 상황으로, 중복 로그인된 다른 기기의 로그인이 풀린다는 안내를 보여준 후 새로운 기기에서의 로그인을 진행합니다.
  - **`false`**: 중복 로그인된 다른 기기가 존재하지 않는 상황으로, 바로 타이머 페이지로 이동합니다.
- 기존 로그인 되어있던 기기(A)를 강제 로그아웃 시킬 때, 해당 기기(A)에서 진행중이던 타이머가 있다면 진행되던 타이머 기록은 삭제됩니다.

## 3. 회원가입 페이지

계정을 생성하는 데 필요한 정보를 입력하고 회원 가입을 합니다.

### 3.1 기본 기능

회원가입에는 총 네 가지 정보가 필요합니다:

- **아이디(Required)**: 이메일 계정을 사용합니다. 중복된 아이디 생성이 제한됩니다.
- **닉네임(Required)**: 서비스에서 사용하고 싶은 자신만의 닉네임을 설정합니다. 중복된 닉네임 생성이 제한됩니다.
- **비밀번호(Required)**: 영문과 숫자 조합으로 이루어진 8자 이상 문자열로 비밀번호를 생성합니다. 입력 시 ⚫️로 값이 가려져서 표시됩니다.
- **비밀번호 확인(Required)**: 먼저 작성한 비밀번호와 같은 값이 입력되었는지 확인합니다. 입력 시 ⚫️로 값이 가려져서 표시됩니다.

- 아이디/닉네임 중복 확인, 네 가지 입력 정보의 유효성 검증, 이용 약관의 동의함 체크까지 모두 이루어져야 '회원가입' 버튼이 활성화됩니다.
- 회원가입에는 `POST /api/signup` API를 사용합니다. 바디(Body)에 유효성 검증을 통과한 아이디(`email`), 닉네임(`nickname`), 비밀번호(`password`), 비밀번호 확인(`confirmPassword`) 정보를 전달합니다.
- '회원가입' 버튼 밑에 '회원이신가요? 로그인 바로가기' 텍스트를 누르면 로그인 페이지로 이동합니다.
- 회원가입에 성공하면 로그인 페이지로 이동합니다.
- 회원가입 성공으로 로그인 페이지로 이동한 후엔 브라우저의 뒤로 가기 기능으로 회원가입 페이지에 다시 이동하는 게 불가능합니다.

### 3.2 유효성 검증

#### 아이디 유효성 검사 규칙

- 아이디는 공백일 수 없습니다.
- 아이디는 이메일 주소 형식을 따라야 합니다.

#### 닉네임 유효성 검사 규칙

- 닉네임은 공백일 수 없습니다.

#### 비밀번호 유효성 검사 규칙

- 비밀번호는 공백일 수 없습니다.
- 총 길이가 8자 이상이어야 합니다.
- 영문과 숫자의 조합으로 구성돼야 합니다.

#### 비밀번호 확인 유효성 검사 규칙

- 비밀번호 확인은 공백일 수 없습니다.
- 비밀번호 확인의 입력 값은 비밀번호의 입력 값과 일치해야 합니다.

#### 유효성 검증 결과에 따른 UI 동작

- 입력 창이 한 번 이상 활성화되고, 입력한 정보가 유효성 검증을 통과하지 못하면 입력창의 경계가 빨간색으로 바뀌고, 아래와 같은 경고 메시지가 입력창 아래에 표시됩니다:
  - **아이디**: 이메일 형식으로 작성해 주세요.
  - **닉네임**: 닉네임을 입력해 주세요.
  - **비밀번호**: 비밀번호는 8자 이상, 영문과 숫자 조합이어야 합니다.
  - **비밀번호 확인**: 비밀번호가 일치하지 않습니다.

### 3.3 아이디/닉네임 중복 확인

- 아이디와 닉네임은 다른 사용자와 중복일 수 없습니다.
- 아이디와 닉네임 입력 창에 유효성 검증을 통과한 값이 입력되기 전에는 '중복 확인' 버튼이 비활성화 상태를 유지합니다.
- 아이디와 닉네임이 입력되고, 유효성 검증을 통과하면 '중복 확인' 버튼이 활성화됩니다.
- '중복 확인' 버튼 클릭 시 각각 `GET /api/signup/check-email` API와 `GET /api/signup/check-nickname` API를 활용해 중복이 있는지 확인하고, 결과를 받아 활용합니다.
- '중복 확인' 버튼을 누르면 서버에 저장된 DB에 중복되는 이메일이 있는지 확인해 결과를 반환합니다.
- 중복 아이디와 닉네임이 있을 시 '이미 사용 중인 이메일(닉네임)입니다'라는 안내가 표시됩니다.
- 아이디와 닉네임에 유효성 검증을 통과한 값을 입력하고 '중복 확인'을 진행하지 않은 상태로 포커스가 벗어나면 입력창에 빨간 outline이 생기고 '중복을 확인해 주세요'라는 안내 문구가 표시됩니다.
- '중복 확인' 결과 아이디와 닉네임이 중복값이 없을 시 입력창 하단에 '사용 가능한 이메일(닉네임)입니다'라는 안내 문구를 표시합니다.

## 4. 프로필 설정 페이지

### 4.1 기본 기능

- 로그인 후 `isFirstLogin` 속성의 값이 `true`일 경우 프로필 설정 페이지로 이동합니다.
- 프로필 설정 페이지는 회원가입 이후 첫 로그인일 때에만 진입합니다.
- 프로필 설정 여부와 관계 없이(프로필 설정을 건너뛰기 하더라도) 두 번째 로그인부터는 바로 타이머 페이지로 이동합니다.

프로필 설정 시 아래 정보들에 대한 값을 수집합니다:

#### 개발 경력

드롭다운 메뉴로 아래 항목들 중 하나의 값을 선택받습니다:

- 경력 없음
- 0 - 3년
- 4 - 7년
- 8 - 10년
- 11년 이상

#### 공부 목적

드롭다운 메뉴로 아래 항목들 중 하나의 값을 선택받습니다:

- 취업 준비
- 이직 준비
- 단순 개발 역량 향상
- 회사 내 프로젝트 원활하게 수행
- 기타(직접 입력)

- '기타(직접 입력)' 선택 시, 기타 공부 목적을 입력받을 수 있는 입력창이 추가로 화면에 표시됩니다.

#### 공부 목표

- 입력 창에 문자열로 공부 목표를 입력받습니다.

#### 공부/사용 중인 기술 스택

- 텍스트 입력 창에 값을 입력해 Chips UI 형태인 태그를 생성하여 값을 추가합니다.
- 기술 스택으로 여러 태그를 선택할 수 있습니다(Multiple).
- 최소한 하나 이상의 기술 스택 태그를 선택해야 합니다.
- 입력 창에 기술 스택 검색어를 입력하면 `GET /api/tech-stacks`로 입력 값(Query String으로 값 전달)을 조회하여 서버에 등록된 적 있는 기술 스택 키워드를 자동완성 UI로 선택 가능하게 보여줍니다.
- 검색어 입력으로 서버에 저장된 기술 스택 정보를 조회할 땐 부분 문자열 검색이 적용되며(e.g. `scr`을 검색하면 `Javascript`와 `Typescript` 모두도 조회됨), 영문 대소문자 구분 없이 탐색합니다(e.g. `scr`를 검색하면 `JavaScript`와 `JavaScript` 모두 조회 가능).
- 입력 값을 통해 원하는 기술 스택 정보가 조회되지 않을 경우, `POST /api/tech-stacks` API를 사용하여 원하는 기술 스택 태그를 생성할 수 있습니다.
- 기술 스택 생성을 위한 POST 요청은 '+ Add New Item'을 클릭하거나, 또는 엔터(Enter) 입력을 통해 보내집니다.
- 생성된 기술 스택 태그 옆에 'X' 표시를 누르면 화면에 있는 태그가 사라지고, 최종 프로필 생성 시 `techStacks` 값으로 보내질 배열에서도 값이 제거됩니다.

#### 프로필 이미지

- 프로필 이미지 업로드 버튼을 누르면 OS에 따라 기본 제공되는 로컬 파일 업로드 기능을 활용해 이미지를 업로드 할 수 있습니다.
- 이미지 업로드 시 `POST /api/file/presigned-url` API를 사용합니다(상세 이미지 업로드 정책은 하단 [4.2 프로필 설정: 이미지 업로드 상세] 부분 참고).
- 클라이언트에서 아래와 같이 업로드 가능한 이미지 파일의 형식과 크기를 제한해줍니다:
  - 업로드하는 이미지의 용량은 **5mb를 초과할 수 없습니다**.
  - 이미지 파일의 형식은 **png, jpg만 업로드 가능**합니다.
- 프로필 정보 저장 시 `profileImage`의 값에는 `POST /api/file/presigned-url` API 요청의 결과로 받은 응답 중 `key` 정보를 전달합니다.
- 프로필 설정에 사용되는 모든 드롭다운 메뉴(개발 경력, 공부 목적 등)는 값을 선택하지 않고 외부를 클릭 시 기존의 값(상태)을 유지한 채로 드롭다운 메뉴가 접힙니다.
- 모든 프로필 설정 값이 입력되면 '저장하기' 버튼이 활성화됩니다.
- '저장하기' 버튼을 누르면 `POST /api/profile` API의 바디(Body)에 입력받은 프로필 값들을 객체로 전달합니다.
- 프로필 정보 저장이 완료되면 별도의 확인 알림 없이 타이머 페이지로 바로 이동합니다.

### 4.2 이미지 업로드 상세

- 클라이언트에서 이미지 업로드 처리 시 **Presigned URL 방식**을 사용합니다.
- `POST /api/file/presigned-url` API로 요청을 보내 서버에서 발급한 AWS S3 버킷의 접근 경로를 응답으로 받습니다.

```json
{
  "presignedUrl": "string", // 서버에서 사전 발급한 S3 버킷 주소
  "key": "string" // 이미지에 접근하기 위한 고유값
}
```

- 서버에서 발급받은 Presigned URL 경로를 활용해 클라이언트에서는 AWS S3 버킷에 직접 이미지를 업로드합니다.
- Presigned URL 경로로 이미지를 업로드 할 땐 PUT 메서드를 사용하고, 요청의 바디(Body)에 업로드 할 이미지 정보를 참조합니다(아래 예시 코드 참조):

```javascript
await fetch(presignedUrl, {
  method: 'PUT',
  headers: {
    'Content-Type': file.type,
  },
  body: file, // 사용자가 직접 업로드 한 이미지 파일 데이터
});
```

- 이미지 파일 업로드 후, `POST /api/file/presigned-url` 요청에서 함께 전달 받은 `key`를 프로필 생성 시(`POST /api/profile`) 바디(Body)의 `profileImage` 속성의 값으로 전달하여 이미지 접근 경로 정보를 서버에 저장해 사용합니다.
- `GET /api/profile` 경로로 프로필 정보를 받아오면 이미지 저장 시 전달한 `key` 정보를 활용해 생성한 이미지 접근 경로가 `profileImage`의 값으로 반환됩니다:

```json
{
  "email": "user@example.com",
  "nickname": "사용자",
  "profile": {
    "career": "1 - 3년",
    "goal": "신입 취업",
    "techStacks": ["JavaScript", "React"],
    "profileImage": "uploads/profile-1753012522898-d0640b60c74ba765.jpg"
  }
}
```

- DevTime의 S3 버킷의 이미지 조회를 위한 공개 URL(`https://dev-time-bucket.s3.ap-northeast-2.amazonaws.com`) 뒤에 전달받은 `profileImage` 경로를 붙여주는 식으로 이미지 정보를 확인 가능합니다(아래 예시 코드 참고):

```javascript
const response = await fetch('/api/profile', {
  headers: {
    Authorization: `Bearer ${token}`,
  },
});

const data = await response.json();

const imageUrl = `https://dev-time-bucket.s3.ap-northeast-2.amazonaws.com/${data.profile.profileImage}`;
// 공개 S3 버킷 URL 뒤에 특정 프로필 이미지 경로 추가하여 접근 링크 생성

// React에서 사용
<img src={imageUrl} alt="프로필 이미지" />;
```

- 서버에서 발행한 Presigned URL 경로는 **5분이 지나면 만료**되고 접근이 불가합니다.
- S3 버킷에 한 번 저장한 이미지는 수정 및 삭제가 불가하며, 프로필 이미지 교체 시에는 새롭게 Presigned URL 경로를 생성하여 이미지를 업로드 한 후 재발급 받은 이미지 접근 경로를 `PUT /api/profile` API로 수정하는 방식으로 처리합니다.

### 4.3 건너뛰기

- 프로필 설정 페이지 하단에 '다음에 하시겠어요? 건너뛰기' 텍스트를 누르면 프로필 설정 단계를 건너뛸 수 있습니다.
- 프로필 설정 건너뛰기 텍스트를 누르면 '프로필 설정을 건너뛸까요?'라는 안내 모달이 나오고, 해당 모달에 있는 '건너뛰기'라는 버튼을 누르면 프로필 설정을 진행하지 않고 서비스를 바로 이용할 수 있습니다.
- 프로필 설정을 건너뛰면 타이머 페이지로 이동합니다.
- 프로필 설정을 건너뛰더라도 [마이페이지]의 [회원정보 수정] 기능을 사용하여 프로필 정보를 입력할 수 있습니다.
- 프로필 정보를 입력하지 않은 상태로 서비스 이용 시 학습 기록이 랭킹에 등재될 수 없습니다.

## 5. 타이머 페이지

### 5.1 기본 기능

- 로그인 후 타이머 페이지에 진입하면 `GET /api/timers` API로 해당 유저가 관리하는 타이머 중 종료되지 않고 계속 유지되고 있는 타이머 기록이 있는지 먼저 확인합니다.
- 타이머를 정지하지 않고 로그아웃, 브라우저 강제 종료, 새로고침 등을 한 경우에는 기존에 재생하던 타이머 기록을 `GET /api/timers` API로 받아와 해당 정보로 타이머 페이지를 초기 세팅합니다(타이머 시간, 오늘의 목표, 할 일 목록 등).
- `GET /api/timers` API의 `splitTimes` 배열에 존재하는 일자별 타이머 경과 시간의 값을 합산하여 화면에 표시돼야 하는 타이머 시간을 연산해 사용합니다.
- 해당 유저가 관리하고 있는 별도의 타이머 정보가 조회되지 않으면 초기 상태로 타이머 페이지를 랜딩합니다.
- DevTime 내 어느 페이지에서든 GNB의 DevTime 로고를 클릭하면 타이머 페이지로 이동합니다.
- 타이머가 재생된 상태로 DevTime 내의 다른 페이지(대시보드, 랭킹, 마이페이지 등)로 이동하더라도 진행 중인 타이머는 동일하게 계속 진행돼야 합니다.
- 다른 페이지로 이동했다가 GNB의 DevTime 로고를 누르면 계속 진행중이던 타이머 정보가 반영되어 화면에 표시되도록 합니다.

### 5.2 시작

- 로그인 한 유저가 타이머 페이지의 재생 버튼(▶︎)을 누르면 타이머를 시작할 수 있습니다.
- 타이머를 재생하기 위해선 '오늘의 목표'와 '할 일 목록'을 작성해야 합니다.
- 타이머 페이지의 재생 버튼(▶︎)을 누르면 '오늘의 목표'와 '할 일 목록'을 작성할 수 있는 모달 창이 나옵니다.
- '오늘의 목표'와 최소 한 개 이상의 '할 일 목록'이 작성돼야 '타이머 시작하기' 버튼이 활성화됩니다. (O)
- 타이머를 시작할 때 작성한 '오늘의 목표'는 타이머 시작 이후에 수정이 불가능합니다.
- '할 일 목록'은 타이머 시작 이후 추가, 삭제, 수정이 가능합니다.
- '오늘의 목표'와 '할 일 목록' 입력 시 글자수 제한은 **30자**입니다(최대 30자 이상 값을 입력할 수 없습니다).
- '오늘의 목표'와 '할 일 목록'을 작성한 후 '타이머 시작하기' 버튼을 누르면 타이머가 동작합니다.
- 타이머를 시작할 땐 `POST /api/timers` API로 서버에 타이머 시작 정보를 전달하며, 요청의 바디(Body)에 모달에 작성한 '오늘의 목표'와 '할 일 목록' 정보를 함께 전달합니다.
- 향후 타이머 시간 데이터 지속 동기화 관리를 위해 응답으로 오는 `timerId` 값을 클라이언트에서 관리해줍니다.
- 타이머를 시작하면 타이머 페이지에 '할 일 목록'을 관리하는 버튼과, 타이머를 '초기화'하기 위한 버튼이 생성됩니다.
- 타이머는 브라우저에서 DevTime 탭이 활성화되지 않았을 때(다른 탭이 브라우저의 최상위에서 동작하고 있을 때)에도 계속 동작합니다.

### 5.3 타이머 데이터 관리

총 타이머 재생 시간에 대한 연산을 클라이언트에서 수행하며, 아래와 같은 사항을 고려해 타이머 시간을 관리합니다:

- 실시간으로 진행 중인 타이머 시간을 클라이언트에서 관리합니다.
- 중간에 타이머를 일시정지 할 경우, 일시 정지한 시간을 제외하고 타이머 시간을 관리합니다.
- 전체 타이머 시간과 함께, 일자 별 진행한 타이머 재생 시간을 쪼개서 관리합니다.

#### 일자별 시간 분리 예시

- 예를 들어, 2025년 3월 3일 23:00에 공부를 시작했고, 3월 4일 01:00에 공부를 끝냈다면 3월 3일에 1시간, 3월 4일에 1시간으로 최종 타이머 재생 시간을 반영합니다.
- 2025년 3월 3일 23:00에 공부를 시작했고, 23:30분부터 24:00까지 타이머를 일시정지했고, 3월 4일 00:00부터 01:00까지 타이머를 다시 재생해 공부를 끝냈다면 3월 3일에 30분, 3월 4일에 1시간으로 타이머 재생 시간을 반영합니다.

- 타이머 재생 시점, 일시 정지, 타이머 종료 시점 등을 고려해 일자 별 타이머 재생 시간을 클라이언트에서 누적 관리하며, 타이머 종료 시 요청 바디(Body) 객체 `splitTimes` 속성에 배열 형태로 일자 별 재생 시간 값을 전달해줍니다:

```json
{
  "timerId": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "splitTimes": [
    {
      "date": "2025-03-03T13:08:38.699Z",
      "timeSpent": 1753000 // ms 초로 환산해서 값 활용
    },
    {
      "date": "2025-03-04T13:08:38.699Z",
      "timeSpent": 3654000 // ms 초로 환산해서 값 활용
    }
  ],
  "review": "stringstringstr",
  "updatedTasks": [
    {
      "content": "string",
      "isCompleted": true
    }
  ]
}
```

- 일자 별로 분리해 전달한 타이머 경과 시간 데이터는 대시보드의 '하루 평균 공부 시간', '요일 별 공부 시간 평균', '공부 시간 바다' 등 특정 일자의 타이머 재생 시간을 구분해서 통계내는 데 활용됩니다.

#### 서버 동기화 (폴링)

- 클라이언트에서 동작되는 타이머의 시간과 서버에서 관리하는 타이머 시간을 주기적으로 동기화하기 위해 **10분에 한 번씩** 타이머 시간 데이터를 서버로 업데이트(폴링; Polling)합니다.
- 타이머 재생 후 10분에 한 번씩 반복적으로 클라이언트에서 관리되는 타이머 시간을 서버로 업데이트합니다.
- 타이머 시간 데이터 업데이트에는 `PUT /api/timers/{timerId}` API를 사용하며, 10분에 한 번씩 클라이언트에서 관리되는 타이머 시간과 관련 정보를 서버로 업데이트 해줍니다.
- 요청의 바디(Body)에는 일자 별로 구분해서 관리하고 있는 타이머 시간 정보(`splitTimes`)를 전달합니다.
- 타이머를 일시정지 할 경우, 일시하는 시점에 `PUT /api/timers/{timerId}` API를 호출하여 해당 시점의 타이머 정보를 업데이트 해주고, 다시 타이머를 시작할 때까지 폴링(Polling) 로직도 멈춥니다.
- 일시정지 후 다시 타이머를 재생할 때부터 폴링(Polling)을 다시 시작합니다. 해당 시점부터 10분에 한 번씩 클라이언트의 타이머 정보를 다시 업데이트 해줍니다.

#### 타이머 초기화

- 타이머 초기화 시, 클라이언트에서 관리되는 타이머 시간 데이터는 초기화되며, `DELETE /api/timers/{timerId}` API 요청을 보내 서버에서 관리하던 타이머 기록도 함께 삭제합니다.

#### 타이머 시간 관리 관련 기획 요약

- **타이머 시간 상태 관리**: 실시간 진행 중인 타이머 시간과, 타이머가 재생된 일자를 고려해 일자 별 진행된 타이머 시간을 구분하여 관리하고, 타이머 업데이트 및 종료 시 배열로 값 전달
- **타이머 시간 상태 중간 업데이트**: 서버와 클라이언트에서 관리하는 타이머 시간을 주기적으로 동기화하기 위하여 `PUT /api/timers/{timerId}` API로 10분에 한 번씩 업데이트
- **타이머 초기화**: 클라이언트에 타이머 상태 초기화 및 `DELETE /api/timers/{timerId}` API로 서버에서 관리되던 타이머 상태도 삭제

### 5.4 일시 정지

- 타이머가 재생되면 타이머를 일시 정지할 수 있는 버튼(⎮⎮)이 활성화됩니다.
- 타이머 일시 정지 버튼을 누르면 클라이언트에서 관리하던 타이머 시간이 일시 정지됩니다.
- 타이머를 일시정지하는 시점에 `PUT /api/timers/{timerId}` API로 요청을 보내 해당 시점까지의 타이머 관리 정보를 서버에 업데이트 해줍니다.
- 타이머를 일시정지 한 시점에는 서버 상태 업데이트(폴링)가 진행되지 않습니다.
- 다시 타이머를 재생하는 시점부터 10분에 한번씩 타이머 상태를 서버로 업데이트(폴링) 하는 작업이 다시 시작됩니다.

### 5.5 초기화

- 타이머가 재생 상태일 때와 일시 정지 상태일 때 모두 사용 가능합니다.
- 진행 중인 타이머를 정식으로 종료하지 않고 값을 초기화할 때 사용합니다.
- 초기화 버튼을 누를 시, `DELETE /api/timers/{timerId}` API 요청을 서버로 보내 서버에서 관리하던 타이머 정보를 삭제해줍니다.
- 타이머를 초기화하면 타이머 시간 정보와, 타이머와 함께 관리되던 '오늘의 목표', '할 일 목록' 정보가 전부 초기화됩니다.

### 5.6 새로고침/페이지 강제 종료/로그아웃 대응

- 타이머 재생 중 정지나 초기화 이벤트를 하지 않고 타이머가 비정상적으로 종료되는 상황이 발생한 경우, 서버에서 가장 최근에 관리되던 타이머 시간과 오늘의 목표/할 일 목록 정보를 살려두고 재로그인 시 해당 정보로 타이머가 세팅되어 있도록 합니다.
- 타이머 정지 또는 초기화를 하지 않은 경우 `DELETE /api/timers/{timerId}` 요청을 서버로 보내지 않게 되고, 서버에는 해당 유저가 관리하던 타이머 정보가 계속 살아있게 됩니다.
- 브라우저 새로고침, 또는 브라우저 강제 종료/로그아웃 등 서비스 이탈 후 서비스 재진입이 될 시 `GET /api/timers` 요청으로 해당 유저가 관리하고 있던 살아있는 타이머 정보가 있는지 조회합니다.
- 만약에, 타이머를 종료하거나 초기화하지 않고, 살아있는 타이머 기록이 있을 경우 해당 유저의 타이머 페이지 초기 상태를 관리하고 있던 타이머 정보(재생되고 있던 타이머 재생 시간, 오늘의 목표/할 일 목록 등)로 타이머 페이지의 데이터 상태를 최신화합니다.
- 타이머 정보를 조회할 때 일자 별 타이머 재생 시간을 누적 관리하던 `splitTimes` 정보도 함께 받아오며, 타이머를 재시작하는 시점을 고려해 일자 별 타이머 재생 시간 기록을 이어서 관리해줍니다.

### 5.7 할 일 관리

- 타이머가 재생되면 '할 일 목록' 버튼이 생성됩니다.
- '할 일 목록' 버튼을 누르면 관리 중인 '할 일 목록'을 확인할 수 있는 모달 창이 확인됩니다.
- 해당 시점까지 추가/수정/삭제를 통해 관리된 할 일 목록이 모달 창에 조회됩니다.
- 해당 시점까지 관리하던 오늘의 목표, 할 일 목표 목록은 `GET /api/study-logs/{studyLogId}`로 조회 가능합니다.
- 타이머 재생 중, 모달 창의 상단 입력 창을 통해 할 일 목록을 새롭게 추가할 수 있습니다.
- 할 일 목록의 아이템(Item)을 클릭하면 할 일이 완료된 것으로 상태가 변경됩니다.
- 할 일 목록에 표시되는 '할 일 수정' 텍스트를 클릭하면 할 일 목록의 아이템이 수정/삭제할 수 있는 상태로 전환됩니다.
- 할 일 목록 아이템 옆에 있는 연필 아이콘을 누르면 해당 항목의 텍스트가 수정 가능한 상태로 바뀌고, 텍스트 수정 완료 후 체크 표시를 누르면 아이템의 내용이 수정됩니다.
- 할 일 목록 아이템 옆에 있는 휴지통 아이콘을 누르면 해당 항목이 삭제됩니다.
- 할 일 목록 아이템의 수정/삭제를 완료한 후 '변경 사항 저장하기' 버튼을 누르면 다시 할 일 목록을 토글할 수 있는 상태로 전환됩니다.
- 체크박스가 되어있는 상태에서 할 일 목록 아이템을 토글하면 해당 항목의 할 일 관리 상태가 완료됨으로 변경됩니다.
- 할 일 완료 여부 체크를 한 뒤 모달 창 하단의 '저장하기' 버튼을 누르면 할 일 관리 상태가 최종적으로 변경됩니다.
- 추가/수정/삭제된 할 일 목록 정보는 `PUT /api/{studyLogId}/tasks` API 요청의 바디(Body)에 배열로 전달합니다.
- 생성된 모달 창은 '취소' 버튼으로만 닫을 수 있고, 모달 바깥 영역을 누르더라도 모달이 꺼지지 않도록 제어합니다.

#### 백그라운드 타이머 재생

- 모달 창에서 할 일 목록을 관리할 때에도 재생 중인 타이머는 동작 상태를 유지하고, 시간은 계속 진행됩니다.

### 5.8 종료

#### 기본 동작

- 타이머 종료 버튼(◼︎)을 누르면 타이머를 종료할 수 있습니다.
- 타이머 종료 버튼(◼︎)을 누르면 관리되던 할 일 목록을 최종으로 확인하고, 학습 회고를 작성할 수 있는 모달 창이 확인됩니다.
- 생성된 모달 창은 '취소' 버튼으로만 닫을 수 있고, 모달 바깥 영역을 누르더라도 모달이 꺼지지 않도록 제어합니다.

#### 할 일 목록 최종 확인

- 타이머 종료 시 확인되는 모달 창을 통해 관리되던 할 일 목록의 완료 여부를 최종적으로 수정할 수 있습니다.
- 타이머 재생 시 할 일 목록 관리와 동일하게 할 일 목록 아이템 추가, 수정, 삭제, 토글(완료 여부 체크)이 가능합니다.
- 최종적으로 관리한 할 일 목록 아이템들의 정보와 상태는 타이머 종료 이후 추후에 수정이 불가능합니다.

#### 학습 회고 작성

- 학습 회고는 일반 텍스트로만 입력이 가능하며, 이미지 첨부 등 다른 미디어 첨부는 불가능합니다.
- 학습 회고는 필수 항목(Required)입니다. 학습 회고는 최소 **15자 이상** 작성이 필수고, 최대 허용 글자수는 **500자**입니다.
- 학습 회고를 15자 이상 작성해야만 타이머 종료를 위한 모달 창의 '공부 완료하기' 버튼이 활성화됩니다.
- 타이머 종료 시점에 작성한 학습 회고는 타이머 종료 후 별도 수정이 불가능합니다.

#### 공부 완료하기

- 타이머 종료 모달의 '공부 완료하기' 버튼을 누르면 타이머가 종료됩니다.
- 타이머 종료 시, `GET /api/timers/{timerId}/stop` API 요청이 필요합니다.
- 최종적으로 수정 완료된 할 일 목록과 학습 회고 기록을 바디(Body)에 포함하여 요청을 전달합니다.
- 클라이언트에서 일자 변경, 일시 정지 등 타이머 재생 관련 데이터를 종합 관리해서 연산한 일자별 학습 시간 데이터를 요청 바디(Body)의 `splitTimes` 속성의 값(배열)으로 함께 전달합니다.
- '공부 완료하기'가 성공적으로 제출되면 타이머 페이지의 타이머는 초기 상태로 돌아갑니다.

## 6. 대시보드 페이지

### 6.1 나의 공부 기록 통계

나의 공부 기록에는 총 5가지 정보가 보여집니다:

#### 누적 공부 시간

- 할 일 목록 체크와 학습 회고 작성까지 완료한 모든 타이머 기록의 공부 시간 총 합을 의미합니다.
- 일시 정지로 멈춰있던 시간은 빼고 누적 공부 시간을 계산합니다.
- 시/분까지만 표시하고, 초는 생략합니다 (e.g. 56시간 34분).

#### 누적 공부 일수

- 할 일 목록 체크와 학습 회고 작성까지 완료한 타이머 기록이 있는 날을 카운팅하여 연속 몇 일 째 공부 중인지를 표시해줍니다.
- 일반 텍스트로 표현됩니다 (e.g. 3일째).

#### 하루 평균 공부 시간

- 타이머 기록이 있는 날 기준으로, 하루에 진행한 타이머 시간(공부 시간)의 평균 시간을 표시해줍니다.
- 일반 텍스트로 표현되고, 시/분까지만 표시됩니다. 초는 생략합니다 (e.g. 8시간 21분).

#### 목표 달성률

- 사용자가 작성한 전체 할 일 목록 대비 완료 체크한 할 일 목록의 비율을 보여줍니다.
- 예를 들어, 1개의 오늘의 목표에 대해 4개의 할 일 목록을 작성했고, 그 중 3개의 할 일을 완료해 토글했다면 목표 달성률은 75%가 됩니다.
- 만약에 다른 학습일의 오늘의 목표에 6개의 할 일 목록을 작성했고, 그 중 5개의 할 일을 완료해 토글했다면, 전체 10개(4개 + 6개)의 할 일 목록 중 8개(3개 + 5개)의 할 일을 토글한 것임으로 목표 달성률은 80%가 됩니다.
- 일반 텍스트로 표현되고, 소수점은 생략합니다 (e.g. 83%).

#### 요일 별 공부 시간 평균

- 타이머 기록이 존재하는 날의 요일 별로 진행된 공부 시간의 평균 값을 보여줍니다.
- 하나의 타이머 기록이 두 개 날짜에 겹쳐있다면, 각 날짜 별 진행된 공부 시간을 구분하여 학습 시간을 관리합니다.
- 예를 들어, 특정 타이머가 2025년 2월 1일 토요일 23:00:00에 시작하여 2025년 2월 2일 일요일 02:00:00에 종료됐다면 2월 1일 토요일에 1시간, 2월 2일 일요일에 2시간이 적용되도록 합니다.
- 날짜별 시간 반영 로직은 클라이언트에서 관리합니다.
- 바(Bar) 차트로 표현하며, X 축에는 요일(월요일-일요일)을, Y 축에는 요일 별 평균 공부 시간을 나타냅니다.
- 차트의 바(Bar)에 마우스를 호버(Hover)하면 해당 요일의 공부 시간 평균 값이 확인됩니다.

#### API 정보

나의 공부 기록표 표현돼야 하는 정보는 `GET /api/stats` API를 통해 서버에서 제공해주는 값을 받아와 사용합니다:

- `consecutiveDays`: 누적 공부 일수
- `totalStudyTime`: 누적 공부 시간
- `averageDailyStudyTime`: 하루 평균 공부 시간
- `taskCompletionRate`: 목표 달성률
- `weekdayStudyTime`: 요일별 공부 시간 평균

### 6.2 공부 시간 바다

- 특정 일자마다 학습한 타이머 시간을 시각화하여 표현해줍니다.
- 특정 일자에 공부한 타이머 기록이 있으면 파란색으로 표시됩니다.
- 특정 날짜에 반영되어 있는 타이머 시간에 따라 파란색의 짙은 정도가 달라집니다:
  - 0-2시간 (가장 옅은 파란색)
  - 2-4시간
  - 4-6시간
  - 6-8시간
  - 8-10시간
  - 10시간 초과 (가장 진한 파란색)
- 열은 주(월), 행은 요일을 나타내며, 특정 주차/특정 요일의 타이머 시간이 해당 위치의 색으로 표현됩니다.
- 바다에는 총 52주의 전체 기록이 표시되며, 화면 width 값이 부족할 경우 가로 스크롤로 전체 일자별 학습 시간 기록이 확인될 수 있도록 합니다.
- 바다는 오늘을 기점으로 1년 동안의 기록을 표현합니다. 오늘 날짜가 속한 주차가 바다에서 가장 오른쪽에 위치하며, 오늘로부터 1년 전까지의 타이머 기록 데이터가 히트맵으로 표현됩니다.
- 바다의 특정 주차-요일 위치에 마우스를 호버(Hover)하면 그날의 공부 시간 정보가 툴팁(Tooltip) 형태로 확인됩니다.
- 툴팁에는 해당 날짜의 공부 시간이 'OO시간 OO분 OO초' 형태로 표시됩니다 (e.g. 공부 시간: 5시간 35분 25초).
- 공부 시간 바다에 표시돼야 하는 일자별 데이터는 `GET /api/heatmap` API를 통해 확인 가능합니다.
- `heatmap`: 배열을 값으로 가집니다. 배열은 일자별 공부 시간(타이머 시간)과 바다에 표시될 색상 레벨(0-5)을 정보로 나타내는 객체들을 값으로 가집니다.

### 6.3 학습 기록

- `GET /api/study-logs` API로 타이머 기록이 존재하는 일자별 전체 학습 기록을 조회할 수 있습니다.
- 응답에는 페이지네이션이 적용되어 있습니다 (한 번에 조회되는 목록 개수는 기본 10개로 설정되어 있습니다).
- 타이머 기록이 존재하는 일자 별 공부 기록을 목록 형태로 나열합니다:
  - **날짜**: 타이머 기록이 존재하는 날짜
  - **목표**: 해당 날짜에 설정한 오늘의 목표
  - **공부 시간**: 일시정지를 제외한 타이머 재생 시간
  - **할 일 갯수**: 타이머 기록에 매칭된 할 일 목록의 총 아이템 갯수
  - **미완료 할 일**: 타이머 기록에 매칭된 할 일 목록 중 완료 체크가 되지 않은 아이템의 개수
  - **달성률**: 타이머 기록에 매칭된 총 할일 목록 아이템 개수 중 완료 체크된 아이템 개수의 비율(소수점 생략)
- 같은 날짜에 두 개 이상의 타이머 기록이 있다면 각각 다른 학습 기록 목록 아이템으로 표시됩니다.
- 특정 날짜의 공부 목표 목록을 누르면 모달이 나옵니다.
- 모달에는 오늘의 목표, 할 일 목록(완료 여부 포함), 한 줄 회고 내용이 포함됩니다.
- `GET /api/study-logs/{studyLogId}` API로 특정 '오늘의 목표'에 해당하는 할 일 목록, 학습 회고 정보를 조회할 수 있습니다.
- 특정 날짜에 이미 진행된 타이머 기록(오늘의 목표, 할 일 목록, 학습 회고)은 수정이 불가능하고, 삭제만 가능합니다 (삭제에는 `DELETE /api/study-logs/{studyLogId}` API를 사용합니다).
- 한 번 삭제된 타이머 기록은 다시 복구될 수 없습니다.

## 7. 랭킹 페이지

### 7.1 랭킹 조회

- 전체 유저의 공부 기록을 바탕으로 1위부터 순위를 차례대로 보여줍니다.
- `GET /api/rankings` API로 유저의 공부 기록을 조회할 수 있습니다.
- 쿼리 파라미터로 정렬 기준, 페이지네이션(무한스크롤 페이지 정보), 조회 정보 개수를 조정할 수 있습니다.
- 서버에서 내려주는 전체 유저의 공부 시간 순위 데이터를 화면에 차례대로 보여줍니다.
- 랭킹에 표시되는 유저의 데이터는 무한스크롤로 처리됩니다.
- 스크롤을 내릴 때마다 순위에 따라 데이터를 순차적으로 불러옵니다.
- 랭킹에 맞게 유저의 프로필 이미지, 닉네임, 학습 시간(누적 학습 시간, 일 평균 학습 시간), 프로필 정보(개발 경력, 개발 공부 목표, 공부 중인 기술 스택)가 목록 형태로 보여집니다.
- 공부 중인 기술 스택은 최대 5개까지만 표시됩니다.

### 7.2 랭킹 정렬 기준 변경

- 랭킹 조회 정렬은 '누적 학습 시간', '일 평균 학습 시간' 두 가지 기준이 사용 가능합니다.
- **'누적 학습 시간'**: 특정 유저의 전체 공부 시간 기록을 전부 합한 값을 의미합니다.
- **'일 평균 학습 시간'**: 타이머 기록이 존재하는 날의 학습 시간의 평균 값을 의미합니다.
- 기본 정렬 기준은 '누적 학습 시간'입니다.
- `GET /api/rankings` API의 쿼리 파라미터로 조회 기준을 전달하여 데이터 조회 정렬 기준을 변경할 수 있습니다.
- 정렬 기준은 API 경로에 `?sortBy={정렬 기준}` 형식으로 쿼리 파라미터를 지정하여 전달합니다:
  - `total`: 총 학습 시간
  - `avg`: 일 평균 학습 시간

## 8. 마이 페이지

### 8.1 프로필 정보 조회

- 마이페이지에서 회원 정보와 개인 프로필 정보를 확인할 수 있습니다.
- **회원 정보**: 이메일 주소, 닉네임
- **개인 프로필 정보**: 공부 목표, 개발 경력, 공부 목적, 개발 스택, 프로필 이미지
- `GET /api/profile` API를 통해 마이페이지에 필요한 프로필 정보를 조회할 수 있습니다.
- 초기 프로필 설정 페이지에서 건너뛰기를 한 경우 공부 목표, 개발 경력, 공부 목적, 개발 스택, 프로필 이미지 등 프로필 정보가 표시되지 않고 기본 Placeholder 텍스트로 대체됩니다.

### 8.2 프로필 정보 수정

- 마이 페이지에 있는 '회원정보 수정' 버튼을 클릭하여 사용할 수 있습니다.
- 수정 기능을 클릭하면 변경 가능한 모든 회원 정보와 프로필 정보가 각 정보에 맞는 타입의 입력 창으로 확인됩니다.
- 초기 프로필 작성 페이지에서 적용됐던 Required와 Multiple 여부, 유효성 검증 여부, 프로필 이미지 업로드 방식 등이 그대로 적용됩니다.
- `GET /api/profile` API로 초기 설정돼있던 프로필 정보와 회원 정보를 불러올 수 있습니다.
- `PUT /api/profile` API로 프로필 변경 사항을 업데이트 할 수 있습니다.
- 필요한 내용 수정을 마친 후 '변경 사항 저장하기' 버튼을 누르면 변경된 값이 적용됩니다.
